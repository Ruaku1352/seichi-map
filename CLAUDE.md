> このファイルは Claude Code が最初に読む「共通の地図」です。
> チームの全員がここを唯一の正（Single Source of Truth）として参照します。

## 0. ハッカソン制約（最重要・先に確認）
- 当日の開発時間: **2日間で合計 7 時間**
- 事前開発: **約1週間 OK**
- **戦略: 事前準備で「動くアプリ」をほぼ完成させ、本番7時間は統合・磨き込み・デモのリハに全振りする。**
- ⚠️ 事前開発の許容範囲はルールで要確認。「コア機能は当日」縛りがある場合は、目玉機能だけ当日実装に回す。

## 1. プロダクト概要
訪日観光客向けの「アニメ聖地巡礼マップ」。
**現在地から徒歩ルートを提示し、その経路上にある聖地のアニメ紹介カードを表示する。**

審査で重視される軸（優先度順）:
1. 完成度・デモの見栄え ← **最優先**
2. アイデア・コンセプトの独自性
3. 社会的インパクト・実用性
4. 技術的な新しさ

## 2. MVP（これだけ完璧に動けば勝ち = ハッピーパス）
1. ユーザーが地図上の現在地（またはデモ用の固定地点）に立つ
2. 近隣の聖地を結ぶ徒歩ルートが地図に線で描かれる
3. 移動（またはルート上の各点）に応じて、その聖地のアニメ紹介カードがポップアップ
4. カードには「アニメ名・シーン・なぜここが聖地か・実際の風景写真・英語の短い紹介文」

**スコープ外（作らない）:** ユーザー登録/ログイン、検索、お気に入り、レビュー、課金、複数言語の完全対応（英語のみでOK）、ネイティブアプリ化。

## 3. 技術スタック（確定）
- **フロント: React（Vite）+ JavaScript**（画面・地図・カード表示）
- **地図: Mapbox GL JS もしくは Google Maps JavaScript API**（※どちらか1つに確定する）
- ルート: 地図サービスの Directions API（徒歩モード）
- **バックエンド: Python（FastAPI）**（AI連携・ルート計算・聖地マッチング）
- データ: 静的 JSON（`seichi_data.json`）。当面はフロントから直接読み込み、必要に応じてバックエンド経由に。
- **AI: Anthropic API（Python バックエンド側から呼び出す）**。紹介文の生成・英訳・トーン調整。
- **ホスティング: フロント = Vercel / バックエンド = Render もしくは Railway**

> 構成メモ: フロント(JS)とバックエンド(Python)は別プログラムとして動き、フロントから API で
> バックエンドを呼ぶ2階建て構成。チームが Python の方が得意なため、主要ロジックは Python 側に置く。
> ブラウザで動く画面まわり（地図・カード）は構造上 JavaScript になる。
>
> ⚠️ 公開先が2つに増えるので、**事前準備のうちにフロント↔バックエンドをつないだ公開テストを必ず1回通すこと**。
> 本番でここが初見だと事故りやすい。

## 4. データ構造（`seichi_data.json` のフィールド定義）
聖地データは**コードから分離**し、データ担当が独立して編集できるようにする。

| フィールド | 型 | 説明 |
|---|---|---|
| `id` | string | 一意なID（例: `uji-eupho-01`） |
| `spot_name_ja` | string | 聖地の名称（日本語） |
| `spot_name_en` | string | 聖地の名称（英語） |
| `anime_title_ja` | string | 作品名（日本語） |
| `anime_title_en` | string | 作品名（英語、公式英題があれば優先） |
| `scene_description` | string | どのシーンの舞台か（紹介文の素材） |
| `intro_short_en` | string | 観光客向けの短い英語紹介文（2〜3文） |
| `lat` | number | 緯度 |
| `lng` | number | 経度 |
| `area` | string | エリア（例: `Uji, Kyoto`） |
| `nearest_station` | string | 最寄り駅 |
| `access_note` | string | 行き方の補足（徒歩◯分など） |
| `photo_url` | string | **実際の風景写真**のURL（アニメ画像は使わない／著作権配慮） |
| `tags` | string[] | 例: `["walkable", "photo-spot", "free"]` |
| `verified` | boolean | データ担当が座標・事実を検証済みか |
| `source` | string | 検証に使った出典URL |

## 5. 役割分担（4人 / Claude Code は1人だけ）
- **Claude Code担当 = 実装エンジン**: フロント・地図・ルート・AI連携を一気通貫で実装。他3人が用意した素材を高速で実装に落とすことに集中。
- **データ担当**: `seichi_data.json` を実在の聖地で埋める。座標と事実は必ず人間が検証（`verified: true`）。
- **デザイン/デモ担当**: 画面ラフ・配色・カードの見せ方を実装前に確定。本番デモの理想ルートの台本とリハ。スライド骨子。
- **統括(PM)/QA/ピッチ**: 仕様書の維持、優先順位の即決、実機QA、発表。Claude Code担当の壁打ち相手。

## 6. コーディング規約 / 約束事
- 聖地データは**必ず JSON 経由**。座標やテキストをコードに直書きしない。
- 1機能ずつ「動く状態」を保つ（常にデモできる状態を死守）。
- アニメのスクショ・キャラ画像はアプリに載せない。**実風景写真・自作イラスト・言葉での描写**に寄せる（著作権配慮）。
- **Anthropic API キーは Python バックエンド側の環境変数（`.env`）に置く。フロント(JS)には絶対に書かない**
  （ブラウザのコードに書くとキーが公開され漏れる）。`.env` は `.gitignore` に入れて GitHub に上げない。
- フロントとバックエンドの責任を分ける: 画面・地図・カード = フロント(JS) / AI・ルート計算・聖地マッチング = バックエンド(Python)。
- フロント↔バックエンドの通信先URL（APIのエンドポイント）も環境変数で切り替えられるようにする（ローカルと本番で変わるため）。

## 8. デモ成功の定義
審査員の前で、**1本の理想ルートを最初から最後まで一度も詰まらずに見せられる**こと。
「地図が動いて、歩くと紹介カードがふわっと出る」——この体験が伝われば勝ち。
